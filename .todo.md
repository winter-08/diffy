# Diffy — Roadmap

## Current State

Diffy is a GPU-accelerated Git diff viewer with syntax highlighting, split/unified views,
word-level inline diffs, GitHub PR loading, and a file tree sidebar. It is **view-only** —
no editing, staging, working directory diffs, or merge support.

---

## Tier 0 — Architectural Prep

Targeted refactors to the four chokepoints that will otherwise rot as features land.
Each is scoped to the minimum needed for the tier 1/2 features that depend on it.
Do each refactor **just before** the feature that needs it, not all at once.

### 0.1 Composable Event Dispatch

**Problem**: All input routing is a single if/else cascade in `app.rs:276-450`.
Every new interactive element (search bar, staging gutter, drag handles) requires
modifying this function and getting precedence right.

**Pattern to steal**: Floem's three-tier event dispatch (GlobalEventCx → RouteCx → EventCx)
with per-component listener registration and capture/target/bubble phases.

**References**:
- Floem dispatch: `.docs/floem/src/event/dispatch.rs` — 3-phase event propagation
- Floem hit-test: `.docs/floem/src/event/path.rs:106-155` — 2-entry LRU cached hit testing
- Floem listeners: `.docs/floem/src/event/listener.rs` — type-keyed per-view listener storage
- GPUI dispatch: `.docs/zed/crates/gpui/src/window.rs:4216-4272` — hit_test → bubble/capture
- GPUI z-order: `.docs/zed/crates/gpui/src/bounds_tree.rs` — R-tree spatial z-order tracking
- Lapce usage: `.docs/lapce/lapce-app/src/editor/view.rs:265-293` — `.on_event()` chaining

**Implementation plan**:

- [ ] Add an `EventListener` type-keyed registry to each component via `HitRegion`
  - Currently `HitRegion` has `{ rect, action, cursor }` — extend with optional event handlers
  - Each component registers its own click/hover/drag handlers during paint
  - Handlers return `EventPropagation::Stop` or `Continue` (Lapce pattern)

- [ ] Replace the if/else cascade in `app.rs` with a dispatch loop
  - Collect hit regions during paint (already happens)
  - On input event: walk hit regions in reverse z-order (already happens)
  - NEW: call the component's registered handler, not a hardcoded match
  - If handler returns `Stop`, done. If `Continue`, try next region.

- [ ] Add `PointerCapture` for drag operations
  - When a component starts a drag (e.g., sidebar resize, hunk edge drag),
    it captures the pointer — all subsequent move/up events go to it
  - Currently sidebar resize is special-cased in `app.rs` — generalize it

- [ ] Move special-case input handling out of `app.rs` into components
  - Sidebar resize → `sidebar_resizer` component owns its drag handler
  - Scrollbar → scrollbar component owns its click/drag handler
  - Text inputs → text input component owns its key handler
  - `app.rs` becomes: "dispatch to hit regions, handle unhandled global keys"

**Do this before**: search overlay (1.3), copy selection (1.4), staging gutter (2.1)

### 0.2 Shell Decomposition

**Problem**: `shell.rs` is 1200 lines doing layout, overlay dispatch, widget rendering,
viewport tracking, and sidebar sizing. Every feature touches this file.

**Pattern to steal**: Lapce's `window_tab()` hierarchy where each area is a separate
function/module, composed at the top level with `Stack::vertical/horizontal`.

**References**:
- Lapce hierarchy: `.docs/lapce/lapce-app/src/app.rs:3298-3362` — window_tab structure
- Lapce workbench: `.docs/lapce/lapce-app/src/app.rs:2184` — panel composition
- Lapce panels: `.docs/lapce/lapce-app/src/panel/view.rs:210-300` — panel containers
- Zed workspace: `.docs/zed/crates/workspace/src/workspace.rs:7882-8180` — render tree
- Zed modal layer: `.docs/zed/crates/workspace/src/modal_layer.rs:77-213` — overlay entity
- Floem layout: `.docs/floem/examples/layout/src/holy_grail.rs` — composable layout
- Floem resizable: `.docs/floem/src/views/resizable.rs:41-78` — resize handle abstraction

**Implementation plan**:

- [ ] Extract `title_bar()` into `src/ui/title_bar.rs`
  - Self-contained: renders logo, repo name, branch, toolbar buttons
  - Receives `&AppState`, returns `AnyElement`

- [ ] Extract `toolbar()` into `src/ui/toolbar.rs`
  - Compare mode selector, layout toggle, renderer toggle, settings
  - Currently inlined in shell.rs — pull out as standalone component

- [ ] Extract overlay dispatch into `src/ui/overlays/mod.rs`
  - Currently shell.rs:112-122 matches on overlay type to render
  - Move the match + sizing logic into `overlays::render_active_overlay()`
  - Each overlay already has its own file — just centralize the dispatch

- [ ] Extract sidebar sizing into `src/ui/sidebar.rs` (or extend existing)
  - `preferred_sidebar_width()` (shell.rs:410-526) is complex and self-contained
  - Move it with the sidebar rendering logic

- [ ] What remains in `shell.rs`: ~200 lines of top-level composition
  - `build_ui_frame()` calls: title_bar + workbench (sidebar + editor) + status_bar
  - Overlay rendered on top via absolute positioning
  - Clean, readable, obvious structure

**Do this before**: any tier 1 feature (reduces merge conflicts)

### 0.3 Extensible Display Layout

**Problem**: `display_layout.rs:rebuild_display_rows()` computes heights for all row types
in one big match. New row types (blame, expandable context, staging indicators, connectors)
mean extending this single function.

**Pattern to steal**: Zed's display_map layer stack where each transformation
(folds, inlays, wraps, blocks) is an independent layer with its own Transform enum.

**References**:
- Zed display_map: `.docs/zed/crates/editor/src/display_map.rs:1-70` — layer architecture doc
- Zed block_map: `.docs/zed/crates/editor/src/display_map/block_map.rs:334-470` — block types
- Zed inlay_map: `.docs/zed/crates/editor/src/display_map/inlay_map.rs:53-94` — Transform pattern
- Floem virtual_stack: `.docs/floem/src/views/virtual_stack.rs:37-52` — dynamic item sizing
- Lapce editor paint: `.docs/lapce/lapce-app/src/editor/` — per-type paint methods

**Implementation plan**:

- [ ] Define a `RowDecoration` trait for extensible row types
  ```
  trait RowDecoration {
      fn height(&self, metrics: &DisplayLayoutMetrics) -> u32;
      fn paint(&self, scene: &mut Scene, rect: Rect, theme: &Theme);
  }
  ```
  - Current row types (FileHeader, HunkSeparator, Context, Modified, Added, Removed)
    become implementations of this trait
  - New types (BlameGutter, ExpandableContext, StagingIndicator) added without
    touching the core layout function

- [ ] Replace the monolithic match in `rebuild_display_rows()` with dispatch
  - Each `RenderRowKind` maps to a `RowDecoration` impl
  - Height computation delegates to the decoration
  - Core function just accumulates y_px and builds the DisplayRow vec

- [ ] Add a `BlockDecoration` for injected rows (Zed's block_map pattern)
  - Blocks are rows injected between existing rows (sticky headers, expand buttons)
  - `BlockPlacement::Above(anchor)` or `Below(anchor)`
  - Blocks have their own height and paint logic
  - This is what makes sticky file headers and expandable context possible

**Do this before**: expandable context (2.2), sticky headers (2.4), blame gutter (3.5)

### 0.4 Curve Rendering Primitive

**Problem**: The renderer supports Rect, RoundedRect, Border, Shadow, Text, Image,
BlurRegion, EffectQuad — but no paths or curves. Connector lines between split
diff panels need bezier curves.

**Pattern to steal**: GPUI's tessellation-based approach — use Lyon to convert bezier
curves to triangle meshes on CPU, then render triangles on GPU.

**References**:
- GPUI Path primitive: `.docs/zed/crates/gpui/src/scene.rs:733-859`
- GPUI PathBuilder: `.docs/zed/crates/gpui/src/path_builder.rs:25-347` — curve building API
- GPUI tessellation: `.docs/zed/crates/gpui/src/path_builder.rs:252-318` — Lyon tessellation
- Floem renderer trait: `.docs/floem/renderer/src/lib.rs:143-159` — stroke/fill abstraction
- Floem backends: `.docs/floem/vello/src/` — Vello scene graph path rendering

**Implementation plan**:

- [ ] Add `lyon` crate dependency

- [ ] Add `PathPrimitive` to the `Primitive` enum in `scene.rs`
  ```
  PathPrimitive {
      vertices: Vec<[f32; 2]>,  // pre-tessellated triangle vertices
      indices: Vec<u16>,         // triangle indices
      color: Color,
      opacity: f32,
  }
  ```

- [ ] Add a `PathBuilder` helper for constructing curves
  - `move_to(point)`, `line_to(point)`, `curve_to(to, ctrl)` (quadratic)
  - `cubic_to(to, ctrl1, ctrl2)` (cubic bezier)
  - `build() -> PathPrimitive` — tessellates via Lyon and returns triangle mesh

- [ ] Add GPU rendering for `PathPrimitive`
  - Simple triangle shader (position + uniform color)
  - Existing quad shader pipeline can be extended or a new simple pipeline added
  - Triangles are the simplest GPU primitive — no new shader complexity needed

- [ ] Add `Scene::path()` convenience method
  - `scene.path(PathPrimitive { ... })` — consistent with `scene.rect()`, `scene.rounded_rect()`

**Do this before**: curved connector lines (2.3)

## Tier 1 — Daily-Drivable

These are the gaps that make diffy unusable for real daily git work.

### 1.1 Working Directory Diff

Compare HEAD vs worktree (unstaged changes) and HEAD vs index (staged changes).
This is the single most common use case for any diff tool.

- [ ] Add `CompareMode::WorkingDirectory` and `CompareMode::StagedChanges`
- [ ] Use libgit2 `diff_index_to_workdir` for unstaged, `diff_tree_to_index` for staged
- [ ] Toolbar selector: "Unstaged" / "Staged" / "All Changes" (like the ref picker but for working state)
- [ ] Auto-refresh on file system changes (debounced, ~500ms)
- [ ] Show untracked files in sidebar with distinct icon/color
- [ ] Empty state when working directory is clean

### 1.2 Hunk Navigation

Jump between changes instead of scrolling through hundreds of context lines.

- [ ] `GoToNextHunk` / `GoToPreviousHunk` actions
- [ ] Default keybindings: `]` next hunk, `[` prev hunk
- [ ] `GoToNextFile` / `GoToPreviousFile` with `n` / `N`
- [ ] Scroll viewport to center the target hunk
- [ ] Hunk counter in status bar: "Hunk 3/17"
- [ ] Wrap around at end/beginning of file (or cross file boundary)

### 1.3 Search in Diff Content

Ctrl+F / Cmd+F to find text within the rendered diff.

- [ ] Search overlay bar (input field + match count + prev/next buttons)
- [ ] Match highlighting across all visible lines
- [ ] Active match distinct color from other matches
- [ ] `Enter` / `Shift+Enter` to cycle through matches
- [ ] Search scoped to current file or across all files (toggle)
- [ ] Case-sensitive / regex toggles
- [ ] Esc to dismiss, persist query until cleared

### 1.4 Copy from Diff Viewport

Users can't copy code from the diff today. Clipboard only works in text inputs.

- [ ] Click + drag to select lines in diff viewport
- [ ] Selection highlighting (painted rectangles over selected rows)
- [ ] `Cmd+C` / `Ctrl+C` copies selected lines as plain text
- [ ] Right-click context menu: "Copy", "Copy with line numbers"
- [ ] Clicking a single line in the gutter selects the whole line
- [ ] Shift+click for range selection
- [ ] Copy strips diff markers (`+`/`-`) by default, option to include them

### 1.5 Keyboard Shortcuts

Every action reachable without mouse.

- [ ] Audit all existing actions and assign default keybindings
- [ ] `]` / `[` — next/prev hunk
- [ ] `n` / `N` — next/prev file
- [ ] `/` — focus sidebar search (already exists)
- [ ] `Cmd+F` — search in diff content
- [ ] `Space` — page down, `Shift+Space` — page up
- [ ] `g` `g` — go to top, `G` — go to bottom (vim-style)
- [ ] `1` / `2` — switch to unified/split view
- [ ] `?` — show keyboard shortcut cheatsheet overlay
- [ ] `Esc` — dismiss any overlay/search, return focus to diff
- [ ] Display shortcut hints in tooltips on toolbar buttons

## Tier 2 — Competitive

These features close the gap with Sublime Merge, GitHub UI, and VSCode.

### 2.1 Hunk / Line Staging

Stage and unstage individual hunks or lines from the diff view.

- [ ] Gutter click to stage/unstage a line
- [ ] Hunk header click to stage/unstage entire hunk
- [ ] Visual distinction: staged lines get a solid accent bar, unstaged get a faded bar
- [ ] Two-diff model (Zed pattern): track HEAD→worktree and HEAD→index simultaneously
- [ ] `DiffHunkSecondaryStatus` equivalent — show partially staged hunks
- [ ] `s` to stage hunk at cursor, `u` to unstage
- [ ] `S` to stage current file, `U` to unstage current file
- [ ] Sidebar shows staging state per file (staged / unstaged / partial)
- [ ] "Stage All" / "Unstage All" toolbar buttons
- [ ] Discard hunk: `d` with confirmation

### 2.2 Expandable Context

Expand context lines between hunks, like GitHub's "Show 20 more lines".

- [ ] Clickable separator between hunks: "Show 20 more lines above/below"
- [ ] Load additional context from the full file content (not just the diff)
- [ ] Smooth expand animation (lines slide in)
- [ ] "Expand all context" option per file
- [ ] Remember expansion state per file during session

### 2.3 Curved Connector Lines (Split View)

Visual lines between left and right panels showing how blocks map.

- [ ] Draw bezier curves from the end of a left-side hunk to the start of the corresponding right-side hunk
- [ ] Color-code by change type: red curves for deletions, green for additions, blue/yellow for modifications
- [ ] Curves rendered in the gutter/gap between the two panels
- [ ] Reduce opacity for non-focused hunks
- [ ] Disable when in unified view

### 2.4 Sticky File Headers

File header stays pinned while scrolling through a long diff.

- [ ] When scrolling past a file header in multi-file view, pin it to the top
- [ ] Show file path + stats in the sticky header
- [ ] Clicking the sticky header scrolls back to file start
- [ ] Stacks if multiple files are visible (max 1 sticky header, replace on scroll)

### 2.5 Scrollbar Change Indicators

Minimap-style markers on the scrollbar showing where changes are.

- [ ] Thin colored ticks on the scrollbar track: green for additions, red for deletions
- [ ] Current viewport position indicator
- [ ] Clicking a tick jumps to that change
- [ ] Works in both split and unified views
- [ ] Optional full minimap column (like VSCode) — lower priority

## Tier 3 — Best-in-Class

These push diffy past every existing tool.

### 3.1 Whitespace Toggle

- [ ] Toolbar toggle: "Hide whitespace changes"
- [ ] Re-diff with whitespace ignored (libgit2 `GIT_DIFF_IGNORE_WHITESPACE`)
- [ ] "Show whitespace" mode: render `·` for spaces, `→` for tabs, `↵` for line endings
- [ ] Per-file or global toggle

### 3.2 Interactive Hunk Edge Dragging

Sublime Merge's signature feature: drag the top/bottom edge of a hunk to adjust context.

- [ ] Hover near hunk boundary shows drag handle cursor
- [ ] Drag up/down to expand or shrink hunk context
- [ ] Live re-render during drag
- [ ] Snap to line boundaries

### 3.3 Open in $EDITOR

- [ ] `e` or `Enter` on a line opens `$EDITOR` (or `$VISUAL`) at that file:line
- [ ] Detect common editors: nvim, vim, code, zed, subl, emacs
- [ ] Configurable in settings
- [ ] Also available via right-click context menu

### 3.4 Commit Flow

Commit without leaving diffy.

- [ ] Commit message input (bottom panel or modal)
- [ ] Show staged changes summary before committing
- [ ] `Cmd+Enter` to commit
- [ ] Amend last commit toggle
- [ ] Post-commit: refresh diff, show toast "Committed: {short_message}"
- [ ] Pre-commit hook support (show hook output on failure)

### 3.5 Blame Gutter

- [ ] Toggle blame annotations in the gutter
- [ ] Show: author, short SHA, relative timestamp
- [ ] Hover popover: full commit message, date, author email
- [ ] Click SHA to view that commit's diff
- [ ] Color-code by recency (recent = warm, old = cool)
- [ ] Only show on one side in split view (the "new" side)

## Future / Aspirational

Things that would make diffy a category-defining tool, but aren't blockers.

- [ ] **Merge tool mode**: accept `$BASE/$LOCAL/$REMOTE/$MERGED`, 3-panel conflict resolution with "accept ours/theirs/both" per conflict, write merged result and exit
- [ ] **Image diffing**: side-by-side image comparison, onion skin, swipe, difference overlay
- [ ] **Stacked diff support**: navigate a stack of dependent PRs (Graphite-style)
- [ ] **Inline comments**: annotate diffs locally (for self-review before pushing)
- [ ] **Directory comparison**: compare two arbitrary directories, not just git refs
- [ ] **Remote diff**: diff across branches on a remote without fetching the full repo
- [ ] **Custom themes**: user-defined color schemes beyond dark/light
- [ ] **Plugin system**: extensible via user scripts or WASM plugins
